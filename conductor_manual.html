Conductor

Full-lifecycle FTTP network design plugin for QGIS. Built for Gigaloch — Scotland's rural fibre ISP.

QGIS 3.22+ Python 3.12 GeoPackage storage EPSG:27700 Gigaloch · Perthshire

What is Conductor?

Conductor is a QGIS plugin that stores an entire FTTP network design — from cabinet to customer — in a single portable GeoPackage file. It provides map-based tools for every stage of the design process: drawing ducts and chambers, placing joints and splitters, importing premises, validating fibre routes, assigning fibre numbers, and generating engineer-ready splice plans.

Designed for Gigaloch's build standard. All naming conventions, splitter cascades, and cable types reflect Gigaloch's real-world network design practices. The 1:4 + 1:8 cascade gives an effective 1:32 split per GPON port on the Calix E7-2 cabinet.

Key Concepts

Understanding how Conductor models an FTTP network.

Two-layer model

Conductor separates the network into two distinct layers, reflecting the real-world build sequence:

🔩
Civil Layer
The physical infrastructure in the ground — ducts, chambers, drop ducts. Built first by the civils team.
🌈
Fibre Layer
The optical network blown through that infrastructure — cables, joints, splitters, bundles. Installed second by the fibre team.

On the map, civil routes are drawn wider (0.7mm navy for spine duct) and fibre routes sit visually inside them as thinner contrasting lines (0.35mm neon pink for cable).

Dock panel layout: list vs grid view

The Design and PIA tabs in the Conductor dock can each be switched between two layouts:

ViewDescription
List viewDefault. Each tool is a labelled button with its name and an (i) help icon, grouped under section headers.
Grid viewA compact icon-only grid (4 columns). Section headers are kept, but each tool becomes a larger icon button with its name shown only as a tooltip on hover. The (i) help buttons are hidden in this view.
Where's the toggle? Look in the top-right corner of the tab bar, next to the PIA tab — a button labelled ▦ Grid view. It applies to whichever tab (Design or PIA) is currently active, and stays put as you switch between them. Click it to switch the active tab between list and grid; each tab remembers its own choice independently (saved via QGIS settings), so Design and PIA can be set differently — the button's checked state updates automatically when you switch tabs to reflect that tab's current view.
In grid view, the active tool is still highlighted (e.g. while mid-way through Digitise Cable), and enabled/disabled state is preserved exactly as in list view — switching views mid-tool is safe.

Network topology

Cabinet (Calix E7-2) └── Spine cable (48F/96F/144F) └── Joint + 1:4 splitter ← SPINE level ├── Distribution cable (12F/24F) → Joint + 1:8 splitter │ └── 8× customer bundles (2F each) ├── Distribution cable → Joint + 1:8 splitter │ └── 8× customer bundles ├── Distribution cable → Joint + 1:8 splitter │ └── 8× customer bundles └── Distribution cable → Joint + 1:8 splitter └── 8× customer bundles
Effective split: 1:4 × 1:8 = 1:32 per GPON port Max customers per port: 32 Calix E7-2 (1 card, 8 ports): 256 customers theoretical max

CRS handling

All data is stored in EPSG:27700 (British National Grid), which gives accurate metre-based measurements across the UK. The QGIS canvas typically displays in EPSG:3857 (Web Mercator) to show the OpenStreetMap basemap — Conductor handles the coordinate transformation automatically on every click.

Creating and opening Conductor projects.

Project summary panel

A status band sits permanently above the Design/PIA tabs, between the postcode search bar and the tab row. It stays visible regardless of which tab is active, giving an at-a-glance read on the overall build:

Refresh on demand. Click the ↻ button (top-right of the panel) to recompute everything. This runs the full route validator synchronously — for Gigaloch's typical build sizes (≤500 premises) this completes in well under a second, so it's deliberately a one-click refresh rather than a continuously-live display. The panel also auto-refreshes once whenever a project is created or opened.
Reading the placeholders. A new project shows -- in every field until the first refresh — this means "not yet calculated", not zero. Once refreshed, a genuine result of zero (e.g. 0 routed on a freshly-imported premises layer) is shown as 0. The "Project summary · updated HH:MM" label shows when the figures were last computed; if the cost estimate or validation step hits an error, it's flagged as "(partial)" with the error available as a tooltip.

Digitise Duct

Click Digitise Duct. Left-click to add vertices along the duct route, snapping to chambers and cabinet. Right-click to finish and save. The duct length is calculated automatically in metres.

Ducts snap to: chambers and cabinet. They must connect between two nodes — never between premises or joints.

Digitise Drop Duct

Click Digitise Drop Duct. First click snaps to a chamber or joint; second click snaps to a premises point. Right-click saves. Drop ducts are always 7mm speedpipe.

Duct sleeve (scaffold bar)

On any duct, the Sleeve field can be set to SCAFFOLD_BAR when either end is embedded in a stream bank (or otherwise protected) instead of ploughing through or boring under — enter the Sleeve Length too. This feeds a dedicated "Scaffold Bar (duct sleeve)" line into the Bill of Materials.

Crossings

Two thin wrappers around Digitise Duct, found in the CROSSINGS section of the Design tab:

ToolBehaviour
Digitise Road CrossingSame as Digitise Duct, but Surface Type is pre-set to ROAD and locked. Permit Ref placeholder hints at S50 road-opening licences.
Digitise Stream CrossingSame as Digitise Duct, but Surface Type is pre-set to WATERCOURSE and locked. Permit Ref placeholder hints at land drainage / watercourse consent.
Crossings are ordinary ducts rows — they slot into the existing chamber-to-chamber duct/cable chain like any other duct segment, so the Route Validator, Fibre Count Calculator, and Fibre Trace need no special handling. Only the Bill of Materials reads surface_type, where each adds its own "Road Crossing" / "Stream Crossing" works-allowance line item.

Edit, Delete & Move

Edit Asset is a single unified tool (edit_assets.py) covering Cabinets/POPs, Chambers, Ducts, Joints, Cables, Drop Ducts, and Bundles — click any asset of any of these types to open its attribute form pre-filled. For a Cabinet/POP this opens the same form as Place/Edit Cabinet/POP, including the Calix equipment counts used by the Cabinet Cost Calculator. Delete Asset — click any asset, confirm the dialog. Move Asset — click an asset, then click its new location; moving a Chamber automatically re-runs the compass-based renumbering.

Fibre Layer

Designing the optical network through the civil infrastructure.

Click Place Joint then click on or near a chamber. The joint snaps to the nearest chamber within 14px and records the chamber_id FK automatically. Fill in the joint type and splitter details.

PIA & Aerial

Pole-mounted aerial network design using BT Openreach Physical Infrastructure Access.

Place Pole

Click Place Pole then click on the map to place a pole point. Fill in the pole_type (e.g. wood, concrete), openreach_ref, and attachment_height_m for the PIA agreement.

Place CBT (aerial closure)

Click Place CBT then click on or near a pole. The closure snaps to the nearest pole within 14px and records the pole reference. CBTs are aerial joints — they behave like underground joints for fibre assignment and splicing, but their joint_type is CBT and closure_type is typically BBUS.

CBTs are joints. Auto-Assign Fibres, Validate Routes, Fibre Trace, and Route Splice Export all treat CBTs identically to underground joints — they appear in the same joints layer and participate in the same BFS traversal.

Place PIA UG Chamber

Click Place PIA UG Chamber then click on the map to place an Openreach underground chamber. This writes a normal chamber record with chamber_type = PIA_UG_CHAMBER, and gets its own SCOT-222-PIAC-NNNN ID series so it's never confused with a Gigaloch-built chamber. Also captures surface_type (Footway / Verge / Carriageway / Private) for Openreach reporting and dig-permit planning.

Digitise PIA Aerial Span

Click Digitise PIA Aerial Span. This is a thin wrapper around Digitise Cable — click each pole/CBT in sequence to draw the aerial span, exactly as you would a normal fibre cable, but the cable's cable_type is automatically set to AERIAL. Fibre Count Calculator and Route Splice Export use this to render the span with a dashed light-blue connector instead of a solid navy one. Because it's the same underlying tool as Digitise Cable, it snaps to the same node types (joints/CBTs and cabinet) and writes the same FK fields — no separate traversal logic is needed anywhere downstream.

Digitise PIA UG Duct

Click Digitise PIA UG Duct to draw a duct running through Openreach's underground infrastructure — typically connecting a Place PIA UG Chamber to a pole or to your own civil network. Snaps to PIA UG Chambers and poles. Writes to the ducts layer with duct_type = PIA_SUBDUCT (the same enum value as the Subduct option in the main Digitise Duct tool), so Edit Asset's duct type dropdown recognises and round-trips it correctly.

Digitise PIA Aerial Drop

Click Digitise PIA Aerial Drop. Identical two-click workflow to Digitise Drop Duct (first click on the CBT, second click on the premises), shown with a light-blue rubber band. drop_type = PIA_AERIAL_DROP is set automatically when the start node is a CBT — no extra configuration needed.

Digitise PIA UG Drop

Click Digitise PIA UG Drop for a drop running from a Place PIA UG Chamber out to a premises. Same two-click workflow (chamber, then premises), written to drop_ducts with drop_type = PIA_UG_DROP.

Chambers carry PIA-specific attributes used for Openreach reporting and the future Cabinet Cost Calculator:

Aerial chains in practice. A typical aerial leg looks like: underground joint → cable (AERIAL) → CBT-001 → cable (AERIAL) → CBT-002 → ... → final CBT with aerial drops to premises. Each CBT passes fibres through to the next, with only the final CBT(s) in the chain actually dropping to customers.
Architecture note. Earlier drafts of this plugin gave aerial spans and drops their own layers and custom BFS traversal logic. That was scrapped — aerial spans and drops are now thin wrappers around the existing Digitise Cable and Digitise Drop tools, just with different default attributes and rubber-band colours. This means the route validator, fibre assignment engine, and trace tools need no PIA-specific code at all; everything just works because it's the same cables/drop_ducts tables and FK chain as the underground network.

Route Validator

Checking every premises has a complete fibre path to the cabinet.

How it works

The validator traces every premises in the layer from its UPRN through the network to the cabinet, following FK references at each step:

premises.uprn → bundle.uprn / bundle.from_joint → joint.joint_id → cable.from_node / cable.to_node → next joint... → cable.pop_id ← terminal condition (CAB in ID)

No geometric snapping is used — the traversal is purely based on stored FK values, making it fast and reliable regardless of map scale.

Running validation

Click Validate Fibre Routes in the TOOLS section. The validator runs in a background thread — QGIS stays responsive. Results stream into the table as each premises is checked.

Result statuses

StatusMeaning
ROUTEDComplete path traced from premises to cabinet
PARTIALTrace started but broke at a specific point — reason given
UNSERVEDNo bundle or drop duct found for this premises

Common PARTIAL reasons

No cable found connected to joint
▾

A joint exists but has no cable digitised from it toward the cabinet. Digitise a cable from this joint using Digitise Cable.

from_joint not found in joints layer
▾

A bundle references a joint ID that doesn't exist. The joint may have been deleted, or the bundle was digitised with an incorrect snap. Re-digitise the bundle snapping to the correct joint.

Drop duct from chamber X but no joint in that chamber
▾

A drop duct originates from a chamber that has no joint placed inside it. Use Place Joint to add a joint to that chamber.

Dead end — no unvisited onward cable
▾

The cable traversal reached a joint with no further cables toward the cabinet. Check the cable topology near this joint — a cable segment may be missing or pointing the wrong direction.

Export results

Click ↓ Export CSV to save the full results table. Columns: UPRN, Address, Status, Reason, Path (full asset chain).

Fibre Assignment

Auto-assigning tube and fibre numbers through the network.

Overview

The fibre assignment tool walks the network from the cabinet outward using a breadth-first traversal. At each node it allocates tube and fibre numbers following IEC 60794, records splitter inputs and outputs, and marks unused fibres as dark storage.

Run validation first. The assignment algorithm assumes the network topology is clean. Run the Route Validator and resolve all PARTIAL results before assigning fibres.

Running assignment

Click Assign Fibre Roles in the FIBRE section. The console window shows the traversal in real time. Once the computation is complete, results are written to the Fibre Assignments layer in the GeoPackage.

Assignment logic

Start at cabinet
Walk outward from the cabinet, following cable from_node/to_node FKs.
At each joint with a splitter
Allocate one fibre as splitter input. Fan outputs to customer bundles. Spare ports marked SPLITTER_OUTPUT_SPARE.
Through-splice to onward cables
Remaining fibres are splice-mapped to the next cable. At branch points, each outbound cable gets the same parallel slice.
End of line — dark storage
At joints with no outbound cables, remaining fibres are marked DARK_STORAGE — coiled and preserved for future use.

Fibre roles

RoleMeaning
THROUGH_SPLICEFibre spliced through to the next cable segment
SPLITTER_INPUTFibre feeding the passive splitter input
SPLITTER_OUTPUTSplitter output port assigned to a customer bundle
SPLITTER_OUTPUT_SPARESplitter output port — unassigned, held for future customer
DARK_STORAGEFibre not yet used — coiled in closure, do not disturb

Fibre Trace

Click any premises to highlight its full route back to the cabinet.

How it works

Fibre Trace runs the same BFS traversal as the Route Validator, but instead of producing a results table it draws the route directly on the map canvas — every drop, cable, and joint between the clicked premises and the cabinet, lit up in a single highlight colour.

Running a trace

Open Fibre Trace
Click Fibre Trace in the FIBRE section. A floating panel appears and the cursor becomes a crosshair.
Click a premises
Click any premises point on the map within snapping distance (18px). The nearest premises is selected automatically.
Review the highlighted route
Every drop, cable, and joint on the path back to the cabinet is highlighted in semi-transparent yellow. The panel shows the address, UPRN, status (ROUTED / PARTIAL), and the full hop-by-hop path.
Clear or retrace
Click Clear Trace to remove the highlight, or click another premises to retrace immediately. Closing the panel (X or Close) also clears the highlight.
Diagnosing faults. If a customer reports a break, click their premises and follow the highlighted route on the map to identify exactly which cable run or joint to investigate in the field. A PARTIAL status shows the route as far as it could be traced before the break.
The highlight is drawn with QGIS rubber bands and is automatically cleared when the dock panel closes, when another Conductor tool is activated, or when the trace panel is closed.

Fibre Count Calculator

Checking that every cable segment is sized correctly for the premises it serves.

How it works

The calculator walks the cable network outward from the cabinet via BFS, counting the unique premises served downstream of each cable segment. It then compares this against the cable's installed fibre count to flag under- or over-specified cables.

Opening the calculator

Click Fibre Count Calculator in the FIBRE section. The table populates immediately from the current GeoPackage — no prerequisites required.

Table columns

ColumnMeaning
Cable IDThe cable segment, with type (SPINE / FEEDER / AERIAL)
From / To NodeThe two joints (or cabinet) this cable connects
Premises DownstreamUnique premises served beyond this cable, counted once even if linked via both a bundle and a drop duct record
Fibres RequiredPremises Downstream × 2 (one working, one spare per premises), plus the recommended minimum standard cable size
Installed Fibre CountThe cable's actual fibre_count from the GeoPackage

RAG status

ColourMeaning
GreenInstalled fibre count is sufficient
AmberWithin 10% of capacity — review before further connections
RedOver capacity — installed fibre count is less than required
Rows are sorted by Premises Downstream descending, so the highest-demand cables (closest to the cabinet) appear first. Click ↻ Recalculate after any design change.
Standard fibre counts checked against: 2, 4, 6, 8, 12, 24, 48, 96, 144, 288.

Route Splice Export

One click, one offline HTML covering every joint between a premises and the cabinet.

How it works

Route Splice Export combines the Fibre Trace BFS with the per-joint splice schedule logic from Splice Plans. Click a premises or a joint anywhere on the map, and it builds a complete route — from the cabinet out to that point — with every joint's splice schedule rendered as a single self-contained HTML document.

Two ways to start a route

Click onResult
A premisesTraces the full route from that premises back to the cabinet (same logic as Fibre Trace)
A jointFinds that joint's position in the network and builds the route from the cabinet out to that joint

Exporting

Open Route Splice Export
Click Route Splice Export in the FIBRE section.
Click a premises or joint
The panel shows the traced route as a chain of joint IDs from cabinet to your selection.
Choose output folder and export
Click Browse to choose a folder (remembered between sessions), then Export Route Splice Plan. You'll be offered the option to open the HTML immediately.

What the export contains

  • A route header with the full chain from cabinet to premises, joint-by-joint
  • One card per joint, in route order (cabinet outward), showing cable in / cable out, and a colour-coded summary bar — splices, drops, pass-through, dark
  • Connector arrows between cards: solid navy for underground cable runs, dashed teal for aerial spans, dashed red for the final drop to the premises — each labelled with cable ID, fibre count, type, and length
  • For any joint with drops, a per-drop table: tube, fibre colour and number, drop/bundle ID, the served premises' address (resolved via UPRN), and drop type (Aerial / Bundle)
Field-ready route pack. Where a single Splice Plan covers one joint, Route Splice Export gives an engineer the whole job — every joint they'll touch on a fault or install, in the order they'll encounter them, with the customer addresses for each drop right there in the table.
New tool — under test. Verified on the SCOT-222 single-cabinet test project. Branch joints (where a route splits) are not yet visually distinguished from the main route. Test thoroughly on multi-branch projects before relying on it in the field.

Bill of Materials

Aggregating quantities and costs across all network layers.

Opening the BoM

Click Bill of Materials in the TOOLS section. The dialog opens immediately and calculates from the current GeoPackage data. It runs on the main thread so the results appear instantly.

Design principle

The civil and fibre layers share the same route but are visually distinct. The wider navy duct line represents the physical pipe in the ground; the thinner neon pink cable line represents the optical fibre blown through it. At a glance, any engineer can see both the civil infrastructure and the fibre network as a single coherent picture.

Topology Rules

The connectivity rules that make fibre tracing reliable.

The four-layer hierarchy

Conductor enforces a strict separation between civil and fibre, and between each level of the fibre hierarchy. Violating these rules will cause the route validator to report PARTIAL routes.

LayerConnects FROMConnects TONever connects to
DuctsChamber or CabinetChamber or CabinetJoints, premises
CablesJoint or CabinetJoint or CabinetChambers, premises
Drop DuctsChamber or JointPremisesCabinet, other joints
BundlesJointPremisesChambers, cabinet

Why this matters

The separation means fibre tracing is reliable at every scale. A cable trace from a premises UPRN walks: bundle → joint → cable → joint → cable → cabinet. Each arrow is a stored FK — no geometric proximity calculations are needed, making the validator fast and unambiguous.

Snapping rules

ToolSnaps to
Digitise DuctChambers, cabinet
Digitise CableJoints, cabinet
Digitise Drop DuctJoints, chambers (start) → premises (end)
Digitise BundleJoints (start) → premises (end)
Place JointChambers (within 14px radius)

Snap radius: canvas.mapUnitsPerPixel() × 14–16 pixels

Double-click phantom event

QGIS fires canvasReleaseEvent twice on a fast double-click. All two-click tools (Drop Duct, Bundle) guard against this using a _last_click_pos check. If you digitise a drop duct and it saves immediately to the wrong location, slow down the two clicks slightly.
